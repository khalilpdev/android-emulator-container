You can run Android emulators on Windows using Docker with WSL2 and nested virtualization, leveraging pre-built images like budtmo/docker-android for development and testing.
Requirements
To run Android in a container on Windows, you need:

Windows 10/11 with WSL2 installed and enabled.
Docker Desktop for Windows (version 4.x or higher) configured to use WSL2.
Nested virtualization enabled in your BIOS/UEFI and in WSL2 (Windows 11 supports this natively).
Sufficient CPU, RAM, and disk space to run the emulator.
Optional GPU support for hardware acceleration (Intel/AMD GPU or NVIDIA with drivers) if you want faster graphics rendering

Run the Container
Example using budtmo/docker-android:
docker run -d -p 6080:6080 -p 5554:5554 -p 5555:5555 \
  -e DEVICE="Pixel 7" \
  --name android-container \
  budtmo/docker-android-x86-13.0
Ports:
6080: Web UI via noVNC
5554: Emulator console
5555: ADB connection
Access the emulator in your browser at http://localhost:6080 to interact with Android

Persistent Storage
By default, the emulator resets on container restart.
To persist data, mount a volume:
-v ~/android_avd:/data
This ensures your AVD and app data are retained across container restarts

Optional GPU Acceleration
For NVIDIA GPUs: add -e GPU="nvidia" --gpus all to enable full OpenGL acceleration.
For Intel/AMD GPUs: use -e GPU="yes" to expose the host GPU to the container.
If no GPU is available, software rendering via SwiftShader is used, which is slower but stable

Reproducible builds: Each container starts with a fresh Android image.

Multiple device profiles: Test apps on different Android versions and screen sizes.
Remote access: Use ADB or noVNC to control the emulator from your host or CI environment
